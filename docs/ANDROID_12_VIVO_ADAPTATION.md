## High priority items for this app

### 1. Background foreground-service start limits
Apps targeting Android 12 cannot freely start FGS while backgrounded; disallowed starts throw `ForegroundServiceStartNotAllowedException`.

**Lumen response**
- Timer path uses `startForegroundServiceSafely()` and always enqueues `TimerReconciliationWorker` first.
- Light / proximity / forced-rest overlay starts wrap `ContextCompat.startForegroundService` in `runCatching` and record failures instead of crashing the caller chain.
- Developer debug overlay starts now use the same refuse-safely pattern.
- Alarm/boot recovery still re-schedules via AlarmManager + WorkManager when a background FGS start is refused.

### 2. PendingIntent mutability required
Every PendingIntent must declare `FLAG_IMMUTABLE` or `FLAG_MUTABLE`.

**Lumen response**
- Alarm, reminder-action, open-app, and notification content PendingIntents all set `FLAG_IMMUTABLE`.
- No mutable PendingIntent path is required by current product flows.

### 3. Explicit `android:exported` for filtered components
Components with intent filters must declare `android:exported`.

**Lumen response**
- Every Activity / Service / Receiver / Provider in Manifest declares `android:exported`.
- Internal alarm/boot/reminder receivers are `exported=false`.
- Open API Activities/Service are intentionally `exported=true` with signature/custom permissions where needed.

### 4. Foreground-service notification timing
Android 12 may delay short FGS notifications unless the notification opts into immediate display or matches special cases.

**Lumen response**
- Shared FGS notification helper calls `setForegroundServiceBehavior(FOREGROUND_SERVICE_IMMEDIATE)`.
- Timer ongoing notification includes an action button and uses service/status categories.

### 5. Untrusted touch occlusion / overlay windows
Touch pass-through overlays (`FLAG_NOT_TOUCHABLE`) are blocked when they occlude other apps untrustedly.

**Lumen response**
- Forced-rest overlay is full-screen and interactive (no `FLAG_NOT_TOUCHABLE`); it is meant to capture interaction during enforced rest.
- Developer debug overlay and toast overlays use `FLAG_NOT_FOCUSABLE` / `FLAG_NOT_TOUCH_MODAL` without `FLAG_NOT_TOUCHABLE`, so they do not create untrusted pass-through touch surfaces.
- Overlay display remains gated on `Settings.canDrawOverlays()`.

### 6. Nested-intent / system-dialog / custom-notification risks
Unsafe nested-intent launch, `ACTION_CLOSE_SYSTEM_DIALOGS`, and fully custom notification layouts have tighter rules.

**Lumen response**
- No nested-intent launch path (`getParcelableExtra` Intent hand-off) found.
- No `ACTION_CLOSE_SYSTEM_DIALOGS` usage found.
- Notifications use standard NotificationCompat templates; no `RemoteViews` / custom content views.

### 7. Motion-sensor sampling rate limit
Motion sensors are capped around 200 Hz unless `HIGH_SAMPLING_RATE_SENSORS` is declared.

**Lumen response**
- Light monitor uses `SENSOR_DELAY_NORMAL`.
- Developer / proximity gates use `SENSOR_DELAY_NORMAL` or `SENSOR_DELAY_GAME` for short samples only.
- No `HIGH_SAMPLING_RATE_SENSORS` declaration and no ultra-high-rate sampling requirement.

## Verification checklist

- Background timer keep-alive: if FGS start is refused, WorkManager reconciliation and alarms still recover session state.
- Notification taps / actions / alarms use immutable PendingIntents and open explicit components.
- Install succeeds with all intent-filter components declaring `exported`.
- Forced-rest overlay appears only with overlay permission and remains interactive (not pass-through).
- Developer debug overlay can start from foreground settings without crashing if background start is refused later.
