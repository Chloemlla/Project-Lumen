# 系统更新检测模板

本文档只描述可复现的技术关键点，不绑定具体源码位置。

## 1. 目标

实现一个“启动自动检查 + 手动检查 + 更新弹窗 + 按平台下载”的更新检测流程。

## 2. 版本基线

- 构建期注入三个核心版本字段：
  - `versionCode`：整数构建号
  - `versionName`：展示版本名
  - `buildTime`：构建时间戳，作为更新判定基准
- 运行时通过编译期常量读取这些值，避免依赖外部配置。
- 更新比较不直接依赖 `versionName`，而是依赖 `buildTime` 与远端发行时间。

## 3. 远端数据源

- 使用 GitHub Releases 作为更新源。
- 拉取 releases 列表后，取第一条作为最新版本。
- 关键字段：
  - `created_at`：发布创建时间，用于更新判定
  - `tag_name`：版本号展示
  - `body`：更新说明展示
  - `assets`：各平台安装包列表

## 4. 更新判定逻辑

- 请求成功后，先判断返回体是否为空或异常结构。
- 将远端 `created_at` 转成秒级时间戳。
- 比较规则：
  - `本地 buildTime >= 远端发布时刻` -> 当前已是最新
  - 否则 -> 存在新版本
- 判定失败时，不中断主流程，只在手动检查场景提示失败。

## 5. 触发方式

- 自动检查：
  - 应用启动后触发
  - 受“自动检查更新”开关控制
  - 仅在非调试环境执行
- 手动检查：
  - 在设置/关于入口显式触发
  - 用户可选择是否关闭自动提醒

## 6. 网络请求约束

- 请求头伪装为移动端 UA。
- 请求时显式传入“未登录账号态”，避免依赖用户登录状态。
- 所有异常捕获后只做日志记录或 toast 提示，不抛到上层。

## 7. 更新弹窗结构

- 发现新版本时展示模态弹窗。
- 弹窗内容建议包含：
  - 版本标签
  - 更新说明正文
  - “查看完整 commit 内容”的跳转入口
  - 下载按钮区
  - 取消按钮
  - 自动检查场景下的“不再提醒”按钮
- 弹窗样式与普通警告对话框一致，避免复杂导航跳转。

## 8. “不再提醒”行为

- 仅在自动检查触发的弹窗中展示。
- 点击后写入持久化开关，关闭后续自动提醒。
- 不影响手动检查。

## 9. 下载包选择

- 根据当前运行平台选择对应安装包：
  - Windows：优先匹配 `zip`、`exe`
  - Linux：优先匹配 `rpm`、`deb`、`tar.gz`
  - 其他平台：直接跳到 GitHub Release 页面
- Android 场景下进一步读取设备 `supportedAbis.first`，按 ABI 匹配资源名。
- 匹配规则同时考虑：
  - 平台关键字
  - 可选扩展名过滤
  - release assets 中的文件名

## 10. 兜底策略

- 找不到匹配资源时，直接打开 latest release 页面。
- 下载行为本质是 URL 跳转，不在客户端内实现安装包解析。

## 11. 持久化配置

- 自动更新开关应持久化。
- 默认开启。
- 关闭后仅影响启动自动检查，不影响手动检查。

## 12. 可复现的最小模板

1. 构建时注入版本号和构建时间。
2. 启动时读取自动更新开关并决定是否调用检查函数。
3. 检查函数请求 releases 列表，取第一项。
4. 用 `created_at` 和本地 `buildTime` 比较。
5. 若有新版本，展示包含说明和下载入口的弹窗。
6. 按平台和 ABI 选择最合适的 asset。
7. 下载失败则回退到 release 总页。

## 13. 复现注意点

- 不要把“版本号字符串”作为唯一判断依据。
- 不要把网络失败当成有新版本。
- 不要把平台下载逻辑写死成单一路径，必须支持 asset 轮询匹配。
- 更新检测必须能被手动触发，且不依赖登录态。
