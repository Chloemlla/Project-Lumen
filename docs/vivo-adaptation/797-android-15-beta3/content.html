1、兼容性

1.1 Stable configuration

说明：

从Android API 1开始，便存在 Configuration 的概念，它提供了 density, orientation, and sizes等信息，应用一般通过 Activity.onConfigurationChanged 获取。

Android API 15之后，一个重大的变化是： Configuration 不再考虑system bars的影响，若应用依据Configuration进行布局计算，须更换为其他更合适的参数，具体如下：

若进行布局计算，建议使用合适的ViewGroup，如 CoordinatorLayout or ConstraintLayout。

若依据Configuration进行 system navbar 的高度判断，建议使用WindowInsets。

若需获取当前应用窗口的size，建议使用computeCurrentWindowMetrics。

影响：

Configuration.screenWidthDp and screenHeightDp 不再受 system bars影响。

Configuration.smallestScreenWidthDp 间接受 screenWidthDp and screenHeightDp 影响。

Configuration.orientation 在接近方形的设备上，间接受screenWidthDp and screenHeightDp的影响。

Display.getSize(Point)间接受Configuration变化的影响。从API level 30便不建议使用此接口。

Display.getMetrics() 不受影响。

如何适配：

若依据Configuration进行 system navbar 的高度判断，建议使用WindowInsets。


1.2 dataSync/mediaProcessing foreground service timeout behavior

说明：

15.0上新引入了两种前台服务类型：dataSync和mediaProcessing。系统对这两种服务的限制是：一个应用的所有dataSync/mediaProcessing类型前台服务一天之内只能前台运行(即调用startForeground)6个小时，当运行满6个小时，服务的onTimeOut方法会被调用。服务的onTimeOut方法被调用后，需要在10s内调用stopSelf方法，否则服务所在进程会发生ANR。

前台运行满6个小时后，该应用的dataSync/mediaProcessing类型前台服务将无法前台运行。

如果应用进入前台，将刷新剩余时间(举个例子，某个应用的dataSync类型前台服务已在后台运行4个小时，剩余时间为2小时，当该应用起了一个Activity进入前台，则该dataSync类型前台服务剩余时间又变为6小时)。

影响范围：

运行在 Android 15 上且使用了dataSync/mediaProcessing类型前台服务的模块。

如何适配：

使用dataSync/mediaProcessing类型前台服务，需重写onTimeOut方法，在onTimeOut方法中调用stopSelf方法，避免模块崩溃。


1.3 OpenJdk变更

1、原始文档

对于以 Android 15 为目标平台的应用，以下变更可能会影响应用兼容性：

（1）字符串格式设置 API 的变化：

在使用String.format() 和 Formatter.format() API 时，现在对参数索引、标志、宽度和精度的验证更为严格：

String.format(String, Object[])

String.format(Locale, String, Object[])

Formatter.format(String, Object[])

Formatter.format(Locale, String, Object[])

例如，使用参数索引 0 时（格式字符串中的 %0）会抛出以下异常：

IllegalFormatArgumentIndexException: Illegal format argument index = 0

（2）对Arrays.asList(...).toArray() 组件类型的更改：

使用 Arrays.asList(...).toArray() 时，所生成数组的组件类型现在是 Object，而不是底层数组元素的类型。因此，以下代码会抛出 ClassCastException：

String[] elements = (String[]) Arrays.asList("one", "two").toArray();

在这种情况下，若要在结果数组中保留 String 的组成部分类型，您可以改用 Collection.toArray(Object[])：

String[] elements = Arrays.asList("two", "one").toArray(new String[0]);

（3）语言代码处理方面的变更：

使用 Locale API 时，希伯来语、意第绪语和印度尼西亚语的语言代码不再转换为过时的形式（希伯来语：iw、意第绪语：ji 和印度尼西亚语：in）。在为其中一个语言区域指定语言代码时，请改用 ISO 639-1 中的代码：yi、Yid1（希伯来语）：yi。heid

（4）随机 int 序列变更：

在 https://bugs.openjdk.org/browse/JDK-8301574 中做出更改后，以下 Random.ints() 方法现在返回的数字序列现在与 Random.nextInt() 方法不同：

Random.ints(long)

Random.ints(long, int, int)

Random.ints(int, int)

Random.ints()

通常，此变更不会导致应用破坏行为，但您的代码不应期望从 Random.ints() 方法生成的序列与 Random.nextInt() 匹配。

1.4 Stopped状态的变化

说明：当长按应用图标并选择“强行停止”之后，该应用就进入STOPPED状态，处于Stopped状态的应用，除非用户主动直接或者间接拉起，除此之外应用是不会响应intent请求的。在 Android 15 上新增了ApplicationStartInfo.wasForceStopped()接口，应用可以此来判断应用是否处于Stopped状态。

另外，当应用在 Android 15 上进入Stopped状态时，系统还会取消所有待处理 intent。这可能会导致一些模块出问题，如定时相关的。

当用户的操作将应用从Stopped状态中移除时，系统将向应用发送ACTION_BOOT_COMPLETED广播，使应用有机会重新注册任何待处理的 intent。各模块应以此做好适配。

影响范围：运行于 Android 15 上的所有应用。


1.5 Edge-to-edge 模式

说明：Edge-to-edge模式是一种应用布局不会避开状态栏/导航栏区域，且应用无法设置状态栏背景颜色，导航栏背景的透明度默认设置为80%的模式。应用可通过设置R.styleable.Window_windowOptOutEdgeToEdgeEnforcement选择是否采用Edge-to-edge模式，安卓15默认采用。

注意：应用采用该模式的话，会对布局有影响，请做好适配工作。

影响范围：targetSdkVersion为35或更高版本的应用。


1.6 使用Spatializer替代Virtualizer

说明：Spatializer 类在 Android 12（API 级别 32）中首次添加，可让应用查询设备上声音空间化的功能和行为。在 Android 15 中，Google废弃了Virtualizer类。如果支持空间化，描述应用希望的内容播放方式请改用：AudioAttributes.Builder.setSpatializationBehavior。

注意：弃用意味着谷歌已结束对这些 API 的官方支持，但开发者仍可继续访问这些 API。

影响范围：使用了Virtualizer类的开发者。

2、用户体验

2.1 弱光增强

说明：弱光增强可以调整预览视频流在弱光条件下的曝光量，创建连续的帧流。可用于：

  • 提供增强的图片预览，以便用户更好地为弱光环境取景。

  • 在光线昏暗的环境下扫描二维码。

检查是否可使用“弱光增强”的方法

开发者可调用如下API： CameraCharacteristics.get(CameraCharacteristics.CONTROL_AE_AVAILABLE_MODES) ，检查返回值是否包含：ON_LOW_LIGHT_BOOST_BRIGHTNESS_PRIORITY。

如果包含，即表明可以使用“弱光增强”。

启用“弱光增强”的方法

CaptureRequest.CONTROL_AE_MODE 设置为 ON_LOW_LIGHT_BOOST_BRIGHTNESS_PRIORITY。

影响范围：相机APP开发者


2.2 HDR 余量控制

说明：Android 15 允许开发者使用setDesiredHdrHeadroom控制 HDR 余量，以在 SDR 和 HDR 内容之间取得平衡。

setDesiredHdrHeadroom方法供应用设置所需的HDR提亮倍数，不同activity可以通过此接口设置不同的提亮倍数。当用户设置的余量超过系统默认的最大余量时，会限制系统最大余量。当用户不设置时，则会使用系统默认最大余量。

左侧屏幕上SDR界面元素的亮度似乎比右侧屏幕的亮度更均匀，这模拟了混合HDR和SDR内容时可能出现的余量问题。通过调整HDR余量，开发者可以在SDR和HDR内容之间更好地平衡。

影响：属新增功能，对已有应用无兼容性风险。


2.3 封面屏幕支持

说明：Android 15 允许应用通过声明 COMPAT_SMALL_COVER_SCREEN_OPT_IN以显示在flip设备的小封面屏幕上。

影响：希望显示在flip设备的小封面屏幕上的APP。属新增功能，对已有应用无兼容性风险。


3、性能

3.1 对 PdfRenderer API 做重大改进

说明：应用可以利用PdfRenderer提供的API整合各种高级功能，例如渲染受密码保护的文件、注解、表单编辑、搜索和通过文案进行选择。支持线性 PDF 优化，以加快本地 PDF 的查看速度并减少资源使用量。

影响：系统中涉及操作pdf的应用，比如笔记等，都可以利用这些接口，快速提供pdf插入批注等上述提到的能力。属新增功能，对已有应用无兼容性风险。


3.2 ApplicationStartInfo API

说明：ApplicationStartInfo 在之前的安卓版本就有了，在安卓15上，这个类增加了更多的应用启动的详细信息，如：进程启动类型（冷启动还是热启动）、应用启动各阶段耗时、应用启动来源（是从广播、content provider、闹钟还是Activity启动的）。

开发者可以通过ApplicationStartInfo更加便捷地获取到上述信息。


3.3 详细的应用大小信息

开发者可以通过传入不同的参数，来分别获取apk文件的大小、dex文件的大小、profile文件的大小、以及apk里so的大小等文件大小占用细节。以便让开发者深入了解应用占用的所有空间。

3.4 SQLite 数据库改进

这个新特性公开了底层 SQLite 引擎的高级功能，这些功能针对应用中可能出现的特定性能问题做了优化。简单来说就是在SQLiteDatabase.java里新增了几个接口，这几个接口可以在以下特定场景下提高性能。

  • 只读延迟事务：在执行只读操作的事务（不包括写操作语句）时，可以使用如下两个方法来执行只读的 DEFERRED 事务：

    1. beginTransactionReadOnly()

    2. beginTransactionWithListenerReadOnly(SQLiteTransactionListener)

    3. 这些事务可以并发运行，并且如果数据库处于 WAL 模式，它们可以与 IMMEDIATE 或 EXCLUSIVE 事务并发运行。

  • 行数和ID：新增了 API，用于检索已更改的行的计数或最后插入的行 ID，而无需发出额外的查询。

    1. getLastChangedRowCount()返回当前事务中的最新 SQL 语句插入、更新或删除的行数；

    2. getTotalChangedRowCount()返回当前连接的相关计数；

    3. getLastInsertRowId()返回要在当前连接上插入的最后一行的行ID。

  • 原始语句：直接调用原始 SQlite 语句，绕过便捷封装容器以及它们可能产生的任何其他处理开销。


4、隐私和安全

4.1 敏感通知保护服务

说明：

Android 15 为了数据安全，设置了被系统判定为敏感类型的通知对象无法被任意监听器获取的限制。

谷歌原生在15上新增android.permission.RECEIVE_SENSITIVE_NOTIFICATIONS权限。

<permission android:name="android.permission.RECEIVE_SENSITIVE_NOTIFICATIONS" android:protectionLevel="signature|role" />

按权限的描述，该权限只有系统级应用可以申请。作用即让应用注册的监听器能感知到敏感通知的真实信息。所有通过NotificationListenerService实现通知监听器的模块都需要去申请此权限，防止出异常。

没有该权限的三方应用注册的监听器，接收到的敏感数据通知的内容会被做如下替换：

标题为：应用名

正文内容为：Sensitive notification content hidden

影响：

所有不具有protectionLevel="signature|role"级别的app都无法接收到敏感通知的真实信息。

何种类型的通知会被识别为敏感信息？

（1）类型：构建通知时，指定通知类型setCategory(“msg/email/social”)

或指定通知样式setStyle(Notification.MessagingStyle/Notification.InboxStyle)

（2）检查内容

必选：通知标题+正文

副选：setSubText设置的辅助信息/设置的Message对象里的相关信息

（3）校验方法

谷歌设置的有相关正则表达式。主要内容为连续的(5-8个)数字/字母字符。并有特殊结尾

（4）简单举例

setCategory(“msg”).setContentTitle(“短信”).setContentText(“验证码 04dd78”)

此种方式构建的通知将符合敏感信息通知。


4.2 屏幕录制检测

Android 15 增加了对应用检测功能的支持，以检测其是否被录制。每当应用在屏幕录制中处于可见或不可见状态之间转换时，系统都会调用回调给到APP。

开发者可通过调用windowManager.addScreenRecordingCallback来注册回调，这样被录屏的应用就能及时获知此时自己正在被录屏，以便该应用开发者应对此场景。


4.3 扩展的 IntentFilter 功能

说明：Android 15 通过 UriRelativeFilterGroup 支持更精确的 Intent 解析。可以使用<uri-relative-filter-group>标签在清单文件里定义这些规则，包括：网址查询参数、网址片段以及屏蔽或排除规则等。这样的好处在于对uri类型的网页，intentfilter声明可以提供更精确的解析规则。

影响范围：需要响应网页类型intent的运行于 Android 15 上的所有应用，请按需适配。


4.4 Health Connect

从 Android 14（API 级别 34）开始，Health Connect 已成为 Android 框架的一部分。而 Android 15 上的本次更新为新数据类型的增加---增加了对健身、营养等领域的新数据类型的支持。

Health Connect 支持多种常见的健康和健身数据类型，其中包括: 活动情况、睡眠质量、营养摄入、体测数据以及重要的体征信息 (比如心率和血压)。

经过用户同意后，开发者们可以通过标准化的架构和 API 调用，安全地从 Health Connect 读取或写入数据。用户对自己的隐私设置持有完全的控制权，Health Connect 为其提供了精细的粒度来查看哪些 App 在某个时刻请求了数据访问。Health Connect 产生的所有数据都被加密存放在设备本地。

开发者可按需使用。


4.5 端到端加密的密钥管理

Android 15 中引入了 E2eeContactKeysManager，这是端到端加密大功能里联系人相关的子模块，旨在与平台联系人应用程序集成，为用户提供集中的方式来管理和验证其联系人的公钥。

E2eeContactKeysManager 管理两种类型的密钥- E2eeContactKey和E2eeSelfKey，通过提供操作系统级 API 来存储加密公钥。

可以理解为两个终端设备之间通信时，一端加密，另一端解密的客户端加解密手段，目的就是防止用户通信内容在经服务器传输时发生泄密。

根据 Android 15 的源码，当前 Android 15 提供了存放这种端到端加密密钥的地方（新的数据库，非联系人数据库），可供app在设备、帐户等维度为联系人插入、更新、删除密钥，并且在对端联系人密钥校验存储后，本端能够获知密钥校验状态。源码并未体现密钥生成、交换相关的逻辑，应由app开发者自行实现，当前框架仅提供了存储密钥的地方。


4.6 使用 fs-verity 保护文件

Android 15 的 FileIntegrityManager 包含一些可让apk开发者利用 Linux 内核中 fs-verity 功能的强大API。

具体来讲，APP 开发者可以调用 FileIntegrityManager类相关方法对 APP 私有路径下的文件开启加密特性，当文件被修改的时候其 fs-verity 摘要值会发生改变，进而能够使得 APP 有一套机制去检测某些重要文件是否被篡改。

请按需适配。
