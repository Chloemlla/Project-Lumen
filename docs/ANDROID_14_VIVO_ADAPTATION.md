## High priority items for this app

### 1. Exact alarms denied by default (targetSdk 33+)
`SCHEDULE_EXACT_ALARM` is not pre-granted for most new installs on Android 14.

**Lumen response**
- Declares `SCHEDULE_EXACT_ALARM`.
- Runtime gate: `AlarmManager.canScheduleExactAlarms()`.
- Scheduling path prefers `setExactAndAllowWhileIdle`, falls back to `setAndAllowWhileIdle` when denied / SecurityException.
- Settings/permission UI routes users to `ACTION_REQUEST_SCHEDULE_EXACT_ALARM`.
- `ExactAlarmPermissionReceiver` resyncs alarms when permission state changes.
- Reminder UX remains useful without exact-alarm grant via inexact fallback + FGS keep-alive when enabled.

### 2. Foreground service types required
Every FGS must declare a type; missing type throws on `startForeground()`.

**Lumen response**
- Timer / light / overlay / debug FGS: `specialUse` with subtype properties.
- Proximity sampling FGS: `camera`.
- No untyped FGS components.
- Not using `dataSync` / `mediaProcessing` long transfer FGS; long transfers are not a product path, so user-initiated data-transfer jobs are N/A.

### 3. Implicit intent / PendingIntent restrictions
Internal components must not rely on package-less implicit intents; PendingIntents need package/component.

**Lumen response**
- Alarm / reminder / notification PendingIntents use explicit receiver/activity classes + package.
- Open-app intents set class + package.
- Open API launch parsing trusts action + sanitized string extras only.
- Share/install streams use `SecureShareIntents` with explicit URI grants.
- Update-available notification content intent reuses the same explicit `openAppPendingIntent()` path.

### 4. Safer dynamic code loading
Dynamically loaded files must be read-only.

**Lumen response**
- Native code loads packaged `System.loadLibrary("lumen_security")` only.
- No runtime-downloaded DEX/SO plugin path.

### 5. Background activity launch tightening
PendingIntent / bindService paths must not silently grant BAL.

**Lumen response**
- Reminder full-screen intents and notification content intents target explicit `MainActivity`.
- No `bindService(... BIND_ALLOW_ACTIVITY_STARTS)` usage found.
- Background timer continuity relies on AlarmManager + typed FGS, not background activity starts.
- Open API "return to caller" only starts the foreign launcher while this Activity is still STARTED; otherwise fail closed.

### 6. Minimum installable targetSdk
Apps with very old targetSdk cannot install on Android 14.

**Lumen response**
- `targetSdk = 37`, far above the floor.

### 7. Partial photo/video access
Media library grants may be partial; apps should not assume full gallery access.

**Lumen response**
- No `READ_MEDIA_IMAGES` / `READ_MEDIA_VIDEO` full-library permission path.
- Sound/backup/template image picks use Storage Access Framework `OpenDocument` (+ persistable grants where needed).

### 8. Process kill API scope
`killBackgroundProcesses()` can only kill the caller's own processes.

**Lumen response**
- No `killBackgroundProcesses` usage found.

## Verification checklist

- Fresh install without exact-alarm grant: reminders still schedule via inexact path; settings can open exact-alarm screen.
- Grant exact-alarm permission: next schedule uses exact alarms and permission-change receiver resyncs.
- Start timer / proximity / light / overlay FGS: no missing-type crash.
- Notification / alarm taps open MainActivity via explicit component.
- Picking custom rest sound / backup file works through SAF without media-library permission.
