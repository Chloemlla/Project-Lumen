# 系统更新检测技术实现指引

这是一套可复用的“发布版自动检查 + 手动触发 + Android 直达下载”模板。核心目标不是只判断版本号，而是同时解决：

1. 启动时自动感知新版本
2. 兼容语义版本号和发布时间两个维度
3. Android 端能按 ABI 选择合适安装包
4. 更新判断和构建元数据保持一致
5. 出错时不打断主流程

## 1. 版本契约

更新检测成立的前提，是“应用内版本信息”和“发布页版本信息”遵循同一套约定。

必须提供这些构建时元数据：

- `versionName`：展示用版本号
- `versionCode`：构建号，建议单调递增
- `buildTime`：UTC 时间戳，作为发布时间兜底判断依据
- `commitHash`：完整提交哈希
- `shortHash`：短哈希，便于发布页标识

运行时读取版本信息时，不要直接依赖硬编码字符串，而是从构建注入值读取。这样做的目的，是让本地构建、CI 构建、正式发布三者在同一套逻辑下工作。

## 2. 发布数据源

更新检测的主数据源应使用发布平台的“最新 Release”接口。返回数据至少要包含：

- `tag_name`
- `published_at`
- `body`
- `html_url`
- `assets`

网络请求需要设置：

- 明确的 `Accept` 和 API 版本头
- 自定义 `User-Agent`
- 有限超时，避免阻塞首屏

如果接口返回非 200，不要中断应用启动，只在手动检查时提示错误。

## 3. 更新判定逻辑

判定不要只看版本号。推荐采用“双条件”：

### 3.1 语义版本比较

先对版本字符串做归一化：

- 去掉前缀 `v`
- 去掉构建后缀
- 去掉元数据后缀

然后按 `major.minor.patch` 逐段比较。只要远端更大，就认为有新版本。

### 3.2 发布时间兜底

有些发布会出现“版本号没变，但 Release 更晚”的情况。此时应再比较 `published_at` 与当前构建时间：

- 当前构建时间从构建注入值读取
- 远端发布时间要转换为 UTC
- 增加一个很小的容忍窗口，避免构建和发布的分钟级误差

只有当“语义版本更新”或“发布时间晚于当前构建”任意一个成立时，才弹出更新提示。

## 4. 启动与手动入口

### 4.1 启动自动检查

应用启动后，在首帧渲染完成再发起检查，避免卡住初始化流程。

自动检查是否启用，使用本地持久化开关保存，默认开启。

### 4.2 手动检查

提供一个设置页入口，用户可以主动触发检查。

手动检查和自动检查的差异：

- 自动检查失败：静默处理
- 手动检查失败：显示错误对话框
- 自动检查发现新版本：直接弹更新提示
- 手动检查发现新版本：同样弹更新提示

## 5. 更新弹窗设计

更新弹窗至少展示：

- 最新 Release 标签
- 当前版本
- 发布时间
- 发布说明

如果判断是“靠发布时间发现的新版本”，要额外说明：

- 即使版本号看起来更低，也应优先升级到更晚发布的构建

按钮策略建议：

- `稍后`
- Android：`查看 Release`、`下载 APK`
- 其他平台：`打开 Release`

## 6. Android 安装包选择

Android 端不要只给一个下载链接。应先读取设备 ABI 列表，再从 Release 资产里匹配对应 APK。

匹配顺序建议：

1. 遍历所有 APK 资产
2. 优先匹配 `arm64-v8a`、`armeabi-v7a`、`x86_64` 等 ABI 标识
3. 找不到精确匹配时，回退到 `universal` 或 `all`
4. 再找不到就打开 Release 页面，让用户手动下载

下载行为不建议在应用内强行拉起安装流程。更稳妥的方式是交给外部浏览器或系统下载器。

## 7. 发布页命名约定

为了让客户端能稳定识别，发布页建议遵循这些约定：

- tag 与应用版本可直接映射
- APK 文件名包含 ABI 信息
- 发布说明正文中保留变更记录
- 如果要做完整性校验，正文中再附带每个 APK 的 SHA256

## 8. 与完整性校验的关系

更新检测和 APK 完整性校验最好共享同一套发布契约。

推荐做法：

- 更新检测：看最新 Release 是否存在
- 完整性校验：看“当前安装包”是否与该 Release 中的哈希匹配

这样可以避免“能检查更新，但无法确认下载包是否官方构建”的断层。

## 9. 最小验证集

这套模板至少要覆盖这些用例：

- 当前版本低于远端版本
- 当前版本高于远端版本
- 版本号相同但发布更晚
- 版本号中含有 `v`、`-hash`、`+metadata`
- 网络超时
- 非 200 响应
- Android 精确 ABI 匹配
- Android 通用包回退

## 10. 复现要点

真正需要复刻的不是某个页面，而是这条链路：

1. 构建时注入版本、构建号、时间和哈希
2. 启动后读取 Release 最新信息
3. 用“语义版本 + 发布时间”双判定
4. 在 UI 中展示差异和发布说明
5. Android 按 ABI 给出可下载 APK
6. 用本地开关控制自动检查
7. 用单元测试锁定版本比较规则

按这七步实现，就能复用同款更新检测模板。
