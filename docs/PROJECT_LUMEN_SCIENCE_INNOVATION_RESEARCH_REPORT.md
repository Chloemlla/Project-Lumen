# Project Lumen 科创研究完整报告

## 一、项目摘要

Project Lumen 是一个面向移动端长时间用眼场景的本地优先型用眼健康与专注辅助系统。项目以 Android 原生 Kotlin 客户端为核心，结合 Rust 后端、MongoDB 数据存储、Admin Dashboard、GitHub Actions 自动发布链路，形成“提醒、监测、统计、个性化、同步、更新、安全防护”一体化的应用原型。

本项目关注的问题是：在学习、办公、阅读、编程和移动设备高频使用场景中，用户容易出现长时间连续近距离用眼、低光环境使用、休息提醒被忽略、专注节奏不稳定、统计反馈缺失等问题。传统定时提醒工具通常只提供固定间隔倒计时，缺少运行可靠性、情境感知、个性化模板、数据闭环和商业化服务能力。Project Lumen 尝试通过本地计时引擎、前台服务、精确闹钟、传感器与相机采样、ML Kit 人脸检测、Room 本地数据、云端备份同步接口和安全通信机制，构建一个可长期运行且可扩展的护眼与专注辅助平台。

项目当前已实现 Android 客户端主体功能、后端基础 API、Admin 管理看板、发布更新策略、安全通信与混淆自防卫基础。后续研究重点将从“功能可用”推进到“质量可验证、隐私可解释、服务可运营、商业化可闭环”。

## 二、关键词

用眼健康；移动端应用；Jetpack Compose；Room；WorkManager；前台服务；ML Kit；Rust；MongoDB；数据同步；隐私保护；应用完整性；R8 混淆

## 三、研究背景与问题提出

### 3.1 背景

智能手机、平板和移动办公设备已经成为学习、工作和娱乐的高频入口。用户在移动端持续阅读、刷题、编程资料查阅、视频会议、短视频观看等场景中，容易形成长时间不间断注视屏幕的行为。仅依赖用户自觉控制休息节奏，往往难以形成稳定习惯。

现有同类工具常见形态包括：

- 简单倒计时或番茄钟应用。
- 系统级健康提醒。
- 蓝光过滤、屏幕色温调节工具。
- 眼保健操或定时弹窗类应用。

这些方案各有价值，但仍存在不足：

- 提醒可靠性不足，后台容易被系统限制。
- 缺少对近距离、眨眼、低光照等场景的综合判断。
- 提醒内容和界面个性化弱，长期使用容易疲劳。
- 统计数据较浅，难以帮助用户理解长期行为。
- 账号、云备份、同步、权益和发布链路不完整。
- 隐私、安全和商业化能力通常在后期补丁式接入。

Project Lumen 的研究目标不是提供医疗诊断，而是围绕“长期连续用眼行为”提供提醒、记录、反馈和个性化辅助，帮助用户降低过长连续用眼和低质量休息的发生频率。

### 3.2 核心问题

本项目将问题拆解为五个研究子问题：

1. 如何在 Android 后台限制条件下保持提醒和计时的可靠性。
2. 如何在不上传原始敏感数据的前提下，利用本地传感器和视觉检测增强护眼提醒。
3. 如何用本地统计、模板和导出能力形成用户可理解的数据反馈。
4. 如何设计客户端、后端和管理系统，使项目具备长期维护和商业化扩展能力。
5. 如何在体积优化、混淆、防篡改和运行稳定之间取得平衡。

## 四、研究目标

### 4.1 总体目标

构建一个可持续迭代的移动端用眼健康与专注辅助系统原型，使其具备稳定运行、本地优先、可解释提醒、数据反馈、隐私保护、安全通信和后续商业化服务能力。

### 4.2 具体目标

- 实现用眼提醒和番茄钟运行态管理，支持暂停、恢复、跳过、休息、通知动作和后台保持。
- 实现近距离、眨眼、低光照、自动亮度、强提醒遮罩等护眼增强能力。
- 建立 Room 本地数据库，保存设置、运行态、统计、模板、权益、计划和备份相关数据。
- 提供多语言 UI、模板系统、统计图表、CSV/PDF/图片导出、本地备份恢复和崩溃报告。
- 建立 Rust 后端，提供账号、权益、同步、备份、遥测、Admin Dashboard 等接口基础。
- 建立 GitHub Actions 自动验证、Release APK 构建、SHA-256 校验和应用内更新策略。
- 建立 HTTPS、证书绑定、请求签名、防重放、Play Integrity、NDK 安全桥和 R8 混淆规则。
- 在不牺牲运行稳定性的前提下减少 release 包体积，并保留 Android、Room、WorkManager、JNI 和 WebView 等运行时边界。

## 五、研究内容与系统方案

### 5.1 Android 客户端

客户端采用原生 Kotlin + Jetpack Compose Material 3 构建，核心模块包括：

- **计时与运行态模块：** 通过 ReminderEngine、PomodoroEngine、RuntimeRepository 和 RuntimeStateEntity 管理提醒与番茄钟状态。
- **后台可靠性模块：** 使用 AlarmManager、Foreground Service、BroadcastReceiver、BootReceiver、WorkManager 和前台通知，提高后台计时恢复能力。
- **护眼检测模块：** 使用前置相机采样、ML Kit Face Detection/Face Mesh、环境光传感器、眨眼概率和距离阈值，触发近距离、干眼和低光提醒。
- **遮罩与通知模块：** 支持全屏强提醒遮罩、自定义 Toast 样式、通知渠道、通知动作和 Android 13+ 通知权限处理。
- **数据与统计模块：** 使用 Room 持久化设置、运行态、每日用眼统计、番茄统计、模板、权益、计划、目标等数据。
- **个性化模块：** 提供内置模板、Pro 模板、颜色和布局配置、系统动态色、图片路径和多语言资源。
- **数据导出与备份模块：** 支持 CSV、统计图片、PDF 月报、本地 JSON 备份和恢复预览。
- **更新模块：** 通过 GitHub Releases 检查版本，下载带 SHA-256 校验值的 APK，并交由 Android 系统安装器处理。
- **安全模块：** 使用安全凭证存储、请求签名、Play Integrity、NDK 安全桥、运行时完整性检查和 R8 混淆规则。

### 5.2 Rust 后端

后端采用 Rust 服务，API 默认挂载在 `/api` 下，主要能力包括：

- 健康检查：`GET /api/health`
- 邮箱登录：`POST /api/v1/auth/email/start`、`POST /api/v1/auth/email/verify`
- 用户信息：`GET /api/v1/me`
- 权益查询：`GET /api/v1/entitlements`
- Google 购买校验入口：`POST /api/v1/purchases/google/verify`
- 同步接口：`GET /api/v1/sync/changes`、`POST /api/v1/sync/push`
- 云备份接口：`POST /api/v1/backups`、`GET /api/v1/backups/latest`
- 遥测接口：`POST /api/v1/telemetry`
- 人脸分析帧记录入口：`POST /api/v1/face-analysis/frames`
- Admin 管理接口：登录、刷新、当前管理员、仪表盘和管理动作。

后端存储使用 MongoDB，集合覆盖用户、登录请求、会话、API nonce、权益、同步变更、备份、遥测、Admin 审计、模板、发布策略和安全 allowlist 等方向。

### 5.3 Admin Dashboard

Admin Dashboard 是零构建静态管理面板，由 Rust 服务托管。它服务于项目运营和研发观测，包括：

- 用户、设备、访问审计和权益变更。
- Google purchase audit、云备份和同步吞吐。
- 崩溃聚合、版本/设备影响和 API health。
- 模板 CMS、视觉模板参数、音频/触觉矩阵和 i18n 分发。
- 匿名宏观遥测、OTA 完整性登记、灰度策略和 HTTP allowlist。

该设计降低了管理前端的构建复杂度，符合项目“少依赖、可复现、可维护”的工程取向。

### 5.4 数据与隐私方案

项目采用本地优先策略：

- 计时、设置、统计、模板和运行态默认保存在本地 Room 数据库。
- 本地备份通过 JSON 导出，恢复前展示摘要。
- 云同步和云备份作为后续 Plus 服务能力，需要登录、用户授权和明确隐私说明。
- 原始摄像头图像不应作为默认上传数据；护眼检测以本地推理和聚合指标为主。
- 遥测需要单独开关，避免与核心提醒功能绑定。

### 5.5 安全与发布方案

安全设计覆盖传输、凭证、请求、运行时和供应链：

- 全站 HTTPS 和明文流量禁用。
- OkHttp 证书绑定能力，支持生产域主备 pin。
- Access Token 与本地凭证通过 Android Keystore + EncryptedSharedPreferences 保存。
- API 请求加入时间戳、nonce 和 HMAC-SHA256 签名。
- 后端保存 nonce 并设置有效期，降低重放攻击风险。
- Play Integrity 用于关键接口的客户端完整性证明。
- 敏感签名密钥读取和基础环境校验下沉 NDK。
- Release 构建启用 R8 和资源收缩，并使用细粒度 keep 规则保护运行时入口。
- GitHub Actions 负责测试、Lint、release APK 构建、SHA-256 产物和发布。

## 六、技术路线

项目技术路线分为五层：

1. **本地核心层：** Kotlin、Compose、Room、DataStore、AlarmManager、Foreground Service、WorkManager。
2. **感知增强层：** Camera、ML Kit Face Detection/Face Mesh、环境光传感器、系统亮度和遮罩服务。
3. **数据闭环层：** 本地统计、目标、模板、导出、备份、崩溃报告和习惯建议。
4. **服务扩展层：** Rust API、MongoDB、账号、权益、同步、备份、遥测、Admin Dashboard。
5. **安全交付层：** HTTPS、证书绑定、请求签名、防重放、Play Integrity、NDK 安全桥、R8 混淆、GitHub Release 更新。

该路线保证基础提醒功能不依赖云端，同时为商业化和多设备能力预留扩展接口。

## 七、创新点

### 7.1 本地优先的综合护眼提醒

项目不是单一倒计时工具，而是将用眼提醒、番茄钟、近距离检测、眨眼提醒、低光照提醒、自动亮度和强提醒遮罩组合为一个本地优先系统。即使无账号、无网络，核心提醒和统计也可运行。

### 7.2 后台可靠性与运行态恢复

项目综合使用前台服务、精确闹钟、WorkManager、开机恢复和持久化运行态，解决移动端后台计时容易中断的问题。

### 7.3 可运营的客户端 + 后端闭环

客户端已具备本地权益、模板、备份、统计和更新能力；后端已具备账号、权益、同步、备份和 Admin Dashboard 基础。相比单机工具，该架构更适合长期运营和商业化扩展。

### 7.4 安全能力前置

项目在原型阶段即纳入 HTTPS、证书绑定、请求签名、防重放、Play Integrity、加密凭证存储、NDK 完整性校验和 R8 规则治理，避免后期安全能力被动补丁化。

### 7.5 体积优化与稳定性平衡

项目不采用全包 keep，而是保留 Android 组件、Room、WorkManager、JNI、WebView JS bridge 和持久化枚举等运行时边界，让普通业务代码继续被 R8 裁剪和优化。这一策略兼顾 release 包体积、混淆强度和运行稳定。

## 八、可行性分析

### 8.1 技术可行性

Android 客户端所用技术均为成熟技术栈：Jetpack Compose、Room、WorkManager、Lifecycle、OkHttp、ML Kit、Android Keystore 等。后端使用 Rust、Axum 风格路由和 MongoDB，适合构建高可靠 API 服务。CI/CD 使用 GitHub Actions，已经覆盖 Android 与 Rust 两端验证路径。

### 8.2 产品可行性

用眼提醒、番茄钟、统计导出和模板个性化具有明确用户场景。Project Lumen 将基础能力免费保留，将高级统计、多计划、Pro 模板、本地完整备份、云同步和云备份作为商业化扩展方向，能降低用户初始使用门槛。

### 8.3 运营可行性

Admin Dashboard 为后续模板、权益、发布、安全 allowlist、崩溃聚合和同步指标管理提供基础。GitHub Release 更新链路使 GitHub 版可以低成本分发，后续可通过 flavor 区分 GitHub、Play、国内和企业渠道。

## 十一、风险与应对

### 11.1 Android 后台限制风险

风险：不同厂商系统对后台服务、精确闹钟、相机后台采样和电池优化限制不同。

应对：保留前台服务、WorkManager、闹钟、开机恢复和权限引导；对不可用权限提供降级路径。

### 11.2 隐私与合规风险

风险：相机、悬浮窗、亮度调节、应用安装、遥测和云同步均涉及敏感权限或敏感数据。

应对：首次启动和设置页明确权限说明；遥测和云同步 opt-in；默认不上传原始摄像头帧；隐私政策明确数据范围。

### 11.3 商业化可信度风险

风险：仅依赖本地手动授权容易被绕过，无法支持退款、撤销和订阅过期。

应对：后续接入真实 Google Play Billing 和服务端验签，以服务端权益快照作为真相来源。

### 11.4 包体积与混淆风险

风险：全包 keep 影响体积和混淆强度；过度裁剪可能导致运行时崩溃。

应对：采用细粒度 R8 规则，保留运行时边界，普通业务代码继续优化；通过 release workflow 验证。

### 11.5 医疗表述风险

风险：护眼应用容易被误解为医疗诊断或治疗产品。

应对：产品和文档使用“提醒”“辅助”“习惯建议”等表述，避免“诊断”“治疗”“预防疾病”等医疗承诺。

## 十二、应用前景

Project Lumen 可面向以下场景扩展：

- 学生群体：学习计时、休息提醒、用眼习惯反馈。
- 办公人群：长时间阅读、会议、文档处理和编程场景。
- 家庭场景：儿童或青少年设备使用提醒，但必须避免监控式设计。
- 企业场景：匿名聚合的团队健康提醒策略和工作节律建议。
- 开源/独立分发场景：GitHub Release 自动更新与可审计构建链路。

项目后续可发展为 Free/Pro/Plus/Team 分层产品：基础提醒免费，高级统计、模板、多计划、完整备份和云同步作为增值能力，团队版提供组织策略和匿名汇总报表。

## 十三、结论

Project Lumen 已完成从单机提醒工具到“客户端 + 后端 + 管理面板 + 发布链路 + 安全机制”的系统化演进。其研究价值体现在移动端后台可靠性、本地感知增强、数据闭环、隐私保护、安全交付和商业化架构整合。

下一阶段应重点完成真实账号体验、服务端权益验签、云同步闭环、隐私权限中心、渠道 flavor 和 GitHub Actions 验证矩阵。完成这些工作后，项目将具备公开发布和持续运营的基础条件。
